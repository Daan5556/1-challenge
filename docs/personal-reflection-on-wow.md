# Personal reflection on way of working

While working on IT projects in the past, I've noticed a common pitfall in my
way of working:

I get so obsessed with the technology, that I forget to document my learning
outcomes.

In practice, that means programming programming microcontrollers, without
documenting my struggles and findings. Or researching network architecture, and
only making notes regarding the technical topic, and not my personal journey. In
the past, that meant that eventough I got results at the end of the project: a
working application with the requirements met. The aftermath, scavenging for
documentation after the fact, was not enjoyable.

This time, I want things to be different. I'm going with a new approach:
documenting as I go. To aid me with this process, I have created an planning
Kanban board planning. The planning items have labels with the learning
outcomes, so if I pick up a new item, I know exactly what outcome I want to
document during the process. This also forces me to plan my project according to
the learning outcomes, without getting carried away by my own nerdiness and
curiosity.

By applying this method, I still can do what I like, which is playing with
computers. I just have an extra guardrail to also have the correct documentation
when playtime is over.
